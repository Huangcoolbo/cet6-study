# CET-6 自动化与 cron 使用评估

## 结论先行

当前可以完成“方案设计”和“文案设计”，但不应立即判定 cron 已可稳定上线。

原因不是技术概念没想清，而是缺少两类验证：
1. 用户真实作息下的触达时段是否合适
2. 提醒发出后，是否真的能形成连续回传与记录

因此当前阶段结论为：
- 可以准备 cron 文案与时间位
- 可以形成上线前检查清单
- 暂不宣称提醒已稳定

## 何时适合启用 cron

满足以下至少 3 项再启用更稳妥：
- 已确定用户大致可学习时段（如 20:30 附近）
- 已有至少 3 天以上真实打卡样本
- 已确定主提醒文案不会过长
- 已明确补救提醒是否需要启用
- 已有固定的回传格式或最小产出标准

## 推荐 cron 方案

### 阶段 1：单次主提醒
- 每天固定 1 次
- 只负责拉起学习
- 不做过多追问

### 阶段 2：主提醒 + 补救提醒
- 当主提醒长期被忽略时再加
- 同日最多 2 次
- 不做深夜提醒

### 阶段 3：周复盘提醒
- 每周固定 1 次
- 用于收拢数据和重排下周重点

## 不建议的做法

- 一开始就设置高频多次催促
- 提醒内容每天都很长
- 在没有真实记录前就自动切换复杂策略
- 用 cron 代替训练记录本身

## 上线前检查清单

- [ ] 已确定主提醒时间
- [ ] 已确定是否需要补救提醒
- [ ] 已确定提醒文案风格
- [ ] 已确定最小回传格式
- [ ] 已确定记录写入位置（如 daily check / task board）
- [ ] 已通过至少一轮人工模拟检查逻辑
