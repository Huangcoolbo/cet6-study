# State Files in `data/index/`

这份文档解释 `data/index/` 里“看起来像状态文件”的内容为什么存在，以及应该怎么理解它们。

很多项目里，状态文件会被当成噪音直接忽略。但在这个仓库里，**有些状态文件本身就是项目流程的一部分**。

---

## 1. 这个目录不只是索引

`data/index/` 不仅保存目录索引和学习跟踪，还承担一部分“项目运行状态”的职责。

因此，这里的文件大致可分为三类：

1. **索引类**
   - 例如资料清单、分类映射、目录说明

2. **跟踪类**
   - 例如任务板、薄弱点跟踪、掌握度跟踪

3. **状态类**
   - 例如提醒流程需要读取或更新的共享状态

---

## 2. `dingtalk-state.json` 是什么？

`dingtalk-state.json` 是钉钉提醒相关流程使用的共享状态文件。

当前理解：
- 它和项目的提醒链路有关
- 它由另一个 OpenClaw 执行流程时读取或更新
- 它保存的是“项目执行连续性”所需要的信息

例如，这类文件通常可能用于：
- 记录最近一次发送过的内容
- 避免重复提醒
- 保持提醒上下文连续
- 给后续执行者一个最小可用状态

这意味着：
- 它**不是普通缓存垃圾**
- 它**不是纯本机临时文件**
- 它**属于项目运行状态的一部分**

所以它应该保留在仓库中，并被纳入同步范围。

---

## 3. 字段级说明：`dingtalk-state.json`

当前文件结构如下：

```json
{
  "recent": [
    "...",
    "..."
  ]
}
```

### 字段：`recent`
- **类型**：字符串数组
- **作用**：保存最近一批已经被钉钉提醒流程选中过的原始条目
- **写入方**：负责钉钉提醒的另一个 OpenClaw / 提醒脚本
- **读取方**：同一条提醒链路，在下次发送前会先读它
- **数据来源**：通常来自 `data/input/` 下的结构化输入文件，如单词、句子、表达、写作句料、翻译句料、听力改写句等

### `recent` 里每一项代表什么？
每一项是一个“已经发送过的训练条目原文”。

根据当前已知提醒脚本，它通常会保存原始输入行，例如：
- `英文\t中文`
- `句子\t释义`

也就是说，这里存的是**被选中过的原始训练项**，不是消息 ID，也不是时间戳。

### 为什么这样设计？
因为提醒流程需要一个简单办法来避免短时间内重复抽到同一条内容。

当前机制是：
1. 从多个 `data/input/` 文件中汇总候选条目
2. 读取 `recent`
3. 尽量从“不在 recent 里的条目”中随机抽取
4. 发送后把本次条目写回 `recent`
5. 只保留最近一小段历史窗口

### 删除这个字段会怎样？
如果 `recent` 被清空或整个文件被删除：
- 提醒流程通常仍然可以继续运行
- 但会失去“最近已发送内容”的记忆
- 短期内更容易重复推送相同条目
- 提醒的连续性和体验会变差

所以它不是致命核心数据，但它是**明显影响提醒质量的协作状态**。

### 为什么它适合进仓库？
因为它：
- 不含凭据本身
- 被另一个 OpenClaw 用于跨轮次执行
- 能帮助多个执行者共享最小状态
- 丢失后虽然不至于系统瘫痪，但会降低项目流程质量

因此它更像“项目运行状态”，而不是“无意义缓存”。

---

## 4. 什么时候状态文件应该进仓库？

一个状态文件应当进入仓库，通常要满足下面至少几个条件：

- 它服务于项目流程，而不是只服务于某台机器
- 它会被其他协作者或其他 OpenClaw 使用
- 丢失它会影响提醒、协作或执行连续性
- 它不包含敏感凭据或私人隐私数据
- 它的存在能被文档解释清楚

---

## 5. 什么时候状态文件不该进仓库？

下面这些通常不该进：

- 仅对单机临时运行有意义
- 可随时重新生成，没有协作价值
- 含敏感信息
- 没有文档解释，也无法判断用途
- 纯噪音文件，只会制造 commit 污染

---

## 6. 对维护者的要求

如果以后新增新的状态文件，最好同时补充以下信息：

- 这个文件是干什么的
- 谁会读它
- 谁会写它
- 如果删掉会影响什么
- 它是否可以安全进入 GitHub

推荐最少做到：
- 在本文件里新增一个说明条目，或者
- 在 README / WORKFLOW 文档中补一段说明

---

## 7. 当前已知状态文件

### `dingtalk-state.json`
- **作用**：钉钉提醒链路共享状态，用于减少重复推送并维持提醒连续性
- **执行者**：另一个 OpenClaw
- **是否应同步**：是
- **是否应进入 GitHub**：是
- **理由**：它属于项目执行流程的一部分，而不是无意义的本地缓存

---

## 8. 维护原则

在这个仓库里，状态文件不按“像不像缓存”判断，而按下面这个标准判断：

> **它是否属于项目运行与协作本身。**

如果属于，而且没有隐私/安全问题，就应该被认真对待、被文档解释、被纳入正确的同步和审查边界。
